# tiny dinos — fully on-chain renderer

Turns the tiny dinos collection (10,000 tokens) into **fully on-chain art and
traits**. The existing per-chain ERC-721 contracts keep their tokens; only their
`baseURI` is repointed to a `web3://` pointer (EIP-4804/6860) at a new on-chain
**renderer**, so `tokenURI(id)` resolves to data stored on-chain instead of IPFS.

The on-chain output is **byte-for-byte identical to the canonical collection** —
same pixels, same traits, same metadata — verified over all 10,000 tokens,
including a **full-resolution 1600×1600 pixel-for-pixel** match.

### Source of truth: the 1600×1600 PNGs

The canonical minted art (the IPFS image in each token's metadata) is the
**1600×1600** PNG. These are *blocky* — every 100×100 region is a single solid
color — so each is really a 16×16-grid image at 100× scale. The renderer encodes
that 16×16 grid (downsampled from the 1600×1600 source) and emits it as a vector
SVG, which scales back to a pixel-perfect 1600×1600.

> The repo also ships `images/.../16x16` PNGs. Those differ from the 1600×1600
> set by ±1 per channel on ~9,795 tokens and by more on 205 snow/night-landscape
> tokens (a pre-existing inconsistency between the two source sets). The on-chain
> renderer follows the **1600×1600** canonical, so it reproduces the minted art
> exactly; it intentionally does **not** match the non-canonical 16×16 PNGs where
> the two disagree.

## How it works

Each dino is a **source-over composite of trait sprites**. The whole collection
reduces to:

- **105 trait sprites** (16×16 RGBA) + **15 unique 1/1 images** — stored once.
- a **per-token record** (5 bytes): the 9 trait value-ids, the render order, the
  minted-on chain, and a 1/1 flag.

Total on-chain data: **~58 KB** (sprites 7.5 KB, offsets 0.5 KB, tokens 50 KB).

Two facts made this exact and compact (both verified against the real assets):

1. **Only 5 distinct layer orders** exist across all 10k tokens (they differ only
   in how `head` / `face` / `eyes` stack). The correct order per token is stored
   in 3 bits.
2. Compositing must use **Pillow's exact integer source-over math**
   (`PRECISION_BITS = 7`). The Solidity `_blend` reproduces it bit-for-bit
   (checked vs PIL over 500k random pairs), so the rendered pixels — including the
   anti-aliased alpha-253/254 edges in "day landscape" backgrounds — match the
   original PNGs exactly.

## Contracts (`contracts/`)

| contract | role |
|---|---|
| `DinoStorage` | holds the three blobs via `SSTORE2`; owner loads them once then `seal()`s — permanently immutable |
| `DinoRenderer` | reads storage, composites the sprites, emits the metadata JSON with an embedded base64 SVG image; `tokenURI` / `metadataJSON` / `imageSVG` |
| `DinoData` | autogenerated constants: category bases, the 5 orders, and all trait label strings |

`DinoRenderer` is `immutable`-wired to a `DinoStorage` and a `currentChain`
string, so one renderer is deployed per chain (eth, avax, bnb, poly, arb, ftm,
opt) to produce that chain's `current-chain` metadata field.

### web3:// wiring (final on-chain step)

Point each live ERC-721's `baseURI` at its renderer:

```
setBaseURI("web3://<rendererAddress>:<chainId>/metadataJSON/")
```

Then `tokenURI(id) = baseURI + id` →
`web3://<renderer>:<chainId>/metadataJSON/<id>`, which resolves (EIP-4804/6860) to
`DinoRenderer.metadataJSON(id)`.

`metadataJSON` returns **raw JSON** — this is exactly what the
[OpenSea metadata standards](https://docs.opensea.io/docs/metadata-standards)
recommend for a web3:// endpoint ("return raw JSON bytes directly… the cleanest
ERC-4804 implementation"). The baseURI deliberately points at `metadataJSON`, not
`tokenURI` — `tokenURI` returns a `data:application/json;base64,…` wrapper, the
"URI inside URI" pattern OpenSea discourages for web3:// (it's kept only for
direct ERC-721 reads). The returned JSON's `image` field is a self-contained
`data:image/svg+xml;base64,…`, so the whole token resolves in a single hop.

## Build & verify

Requires Python 3 with `Pillow` and `pycryptodome`.

```bash
python3 build/build.py
```

This runs the whole pipeline and asserts **10,000/10,000 pixel-exact** and
**70,000/70,000 metadata-exact** (10k × 7 chains):

1. `solve_orders.py` — derive each token's exact render order
2. `extract.py` — encode the on-chain blobs (`build/out/`)
3. `gen_solidity.py` — generate `contracts/src/DinoData.sol`
4. `gen_fixtures.py` — generate the differential-test fixtures
5. `reference_render.py` — full pixel + metadata verification

### Solidity differential test

```bash
cd contracts
forge install foundry-rs/forge-std   # one-time
forge test                           # fast: sample + per-chain
forge test --match-test test_AllTokens -vv   # full 10k via keccak hashes
```

The test deploys the contracts, loads the real blobs, and asserts the renderer's
output equals the Python fixtures — which are themselves proven exact against the
original collection.

> The Python EVM harness runs the actual compiled contracts on `py-evm` and
> compares directly against the canonical repo PNGs — no fixtures in between:
> `build/evm_comprehensive.py` runs all four checks — dinos at the 16-grid **and
> full 1600×1600** (vs `images/dinos/1600x1600/original`, including the
> snow/night-landscape outliers), every trait sprite (`traitRGBA` vs
> `images/traits/1600x1600`), and `tokenURI` JSON (base64-decoded vs
> `metadata/eth`). `EVM_STEP=1` covers all 10k.

### Upscaling / resolution

The on-chain image is an **SVG**, so it is resolution-independent: a marketplace
renders it at any size as crisp solid blocks. Rasterized at 1600×1600 it is
**pixel-for-pixel identical** to the canonical `images/dinos/1600x1600/original`
PNGs — verified for all 10,000 tokens at the 16-grid and on a full-resolution
sample (including the snow/night-landscape tokens that previously diverged).
